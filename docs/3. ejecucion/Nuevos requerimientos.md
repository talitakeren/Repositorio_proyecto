## 2. Cumplimiento de Requerimientos en la Rama `development`

La rama **development** implementa y amplía los requerimientos definidos inicialmente, incorporando funcionalidades adicionales que fortalecen la gestión académica, la generación de horarios y la administración del sistema.

| ID     | Requerimiento                              | Implementación en development                                                                                                                                                                  |
| ------ | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| RF-01  | Registro de cursos                         | El sistema debe permitir registrar, editar, consultar y eliminar cursos, incluyendo la definición del tipo de aula requerida y el estado activo/inactivo de cada curso.                        |
| RF-02  | Registro de docentes                       | El sistema debe permitir gestionar docentes, registrar su especialidad, los cursos que pueden dictar y su disponibilidad según las franjas horarias oficiales.                                 |
| RF-03  | Registro de aulas                          | El sistema debe permitir administrar aulas considerando su tipo, estado operativo y disponibilidad para la programación académica.                                                             |
| RF-04  | Registro de estudiantes                    | El sistema debe permitir gestionar estudiantes, asociarlos a cuentas de usuario y mantener información académica relevante para su matrícula.                                                  |
| RF-05  | Validación de matrícula por prerrequisitos | El sistema debe validar automáticamente los prerrequisitos de cada curso e informar los requisitos pendientes antes de completar la matrícula.                                                 |
| RF-06  | Validación de límite de créditos           | El sistema debe controlar los límites mínimos y máximos de créditos permitidos y mostrar el total acumulado durante el proceso de matrícula.                                                   |
| RF-07  | Generación automática de horarios          | El sistema debe generar horarios automáticamente mediante un motor CSP que considere disponibilidad de docentes, capacidad y tipo de aulas, restricciones académicas y matrículas confirmadas. |
| RF-08  | Visualización de horario por estudiante    | El sistema debe permitir que cada estudiante consulte su horario académico personalizado.                                                                                                      |
| RF-09  | Visualización de horario por docente       | El sistema debe permitir que cada docente visualice su carga horaria semanal asignada.                                                                                                         |
| RF-10  | Visualización de horario por aula          | El sistema debe permitir consultar la ocupación y disponibilidad de cada aula por franja horaria.                                                                                              |
| RNF-01 | Tiempo de generación ≤ 10 s                | El sistema debe generar horarios en tiempos reducidos mediante optimizaciones y filtros previos que mejoran el rendimiento del motor CSP.                                                      |
| RNF-02 | Tiempo de respuesta API ≤ 2 s              | El sistema debe proporcionar respuestas rápidas mediante una arquitectura optimizada y manejo eficiente de las solicitudes.                                                                    |
| RNF-03 | Interfaz intuitiva                         | El sistema debe ofrecer interfaces diferenciadas por rol y procesos guiados que faciliten la navegación y el uso de las funcionalidades.                                                       |
| RNF-04 | Protección de datos                        | El sistema debe proteger la información mediante autenticación JWT, cifrado de contraseñas y control de acceso basado en roles.                                                                |
| RNF-05 | Estructura modular del código              | El sistema debe mantener una arquitectura modular y organizada que facilite el mantenimiento y la reutilización de componentes.                                                                |
| RNF-06 | Escalabilidad CSP                          | El sistema debe permitir ampliar y configurar las restricciones del motor CSP sin afectar su funcionamiento principal.                                                                         |
| RNF-07 | Compatibilidad con navegadores             | El sistema debe garantizar su correcto funcionamiento en los principales navegadores modernos compatibles con aplicaciones web SPA.                                                            |
